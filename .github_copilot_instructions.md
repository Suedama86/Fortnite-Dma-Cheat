# GitHub Copilot Instructions for Fortnite DMA Project

## Project Context

This is a Direct Memory Access (DMA) project for memory introspection and analysis. The project supports both traditional hardware-based DMA and modern virtualized environments using memflow backend.

## Code Style Guidelines

### C++ Standards
- Use C++17 standard
- Prefer smart pointers over raw pointers
- Use RAII principles for resource management
- Follow consistent naming conventions:
  - Classes: `PascalCase` or `c_snake_case` (match existing)
  - Variables: `snake_case`
  - Constants: `UPPER_SNAKE_CASE`
  - Functions: `snake_case`

### Memory Safety
- Always validate memory addresses before access
- Use size_t for memory sizes and array indices
- Implement proper error handling for memory operations
- Use const-correctness throughout

### DMA-Specific Patterns
- Batch memory operations when possible for performance
- Implement scatter-gather operations for efficiency
- Use appropriate flags for memory access (NOCACHE, NOPAGING, etc.)
- Always check return values from memory operations

## Architecture Patterns

### Backend Abstraction
The project uses a backend abstraction pattern:
```cpp
// Prefer backend-agnostic interfaces
class MemoryInterface {
public:
    virtual bool ReadMemory(uint64_t address, void* buffer, size_t size) = 0;
    virtual bool WriteMemory(uint64_t address, const void* buffer, size_t size) = 0;
};
```

### Error Handling
Implement robust error handling:
```cpp
// Use result types or exceptions consistently
template<typename T>
class Result {
    T value;
    bool success;
    std::string error_message;
};
```

### Performance Optimization
- Use memory pools for frequent allocations
- Implement caching for frequently accessed data
- Batch operations to reduce overhead
- Use asynchronous operations where appropriate

## Best Practices Summary

1. **Always validate inputs** - Check addresses, sizes, and parameters
2. **Use backend abstraction** - Write code that works with multiple backends
3. **Implement proper logging** - Use structured logging for debugging
4. **Handle errors gracefully** - Provide meaningful error messages
5. **Optimize for performance** - Batch operations and use caching
6. **Maintain compatibility** - Support both Windows and Linux where possible
7. **Document thoroughly** - Include usage examples and parameter descriptions
8. **Test comprehensively** - Unit tests, integration tests, and performance tests

## Common Anti-Patterns to Avoid

- **Hardcoded memory addresses** - Use signature scanning instead
- **Blocking operations in main thread** - Use async operations
- **Memory leaks** - Always use RAII or smart pointers
- **Magic numbers** - Use named constants
- **Global state** - Prefer dependency injection
- **Overly complex inheritance** - Favor composition over inheritance

This document should guide GitHub Copilot to generate appropriate code suggestions that align with the project's architecture and requirements.
